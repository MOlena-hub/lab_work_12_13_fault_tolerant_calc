## Практична робота №12-13: Обробка сигналів та відновлення після збоїв у багатопоточних програмах

**Варіант №5.** Напишіть багатопоточну програму, яка виконує критичну обчислювальну задачу, і у випадку SIGSEGV відновлюється з останньої збереженої контрольної точки.

## Опис проєкту  
Цей проєкт присвячений розробці відмовостійкого (**fault-tolerant**) програмного забезпечення в середовищі Linux. Основна увага приділяється перехопленню критичних помилок сегментації пам'яті (`SIGSEGV`) у багатопотоковому середовищі та використанню нелокальних переходів для повернення програми до стабільного стану (контрольної точки) без її повного завершення.

## Якість коду та форматування  
Для забезпечення надійності та відповідності системним стандартам використано:  
* **Стандарт іменування**: Папка проєкту `lab_work_12_13_fault_tolerant_calc` та імена змінних у стилі **snake_case**.
* **Thread-Local Storage (TLS)**: Використання ключового слова `__thread` для ізоляції контексту відновлення кожного окремого потоку.
* **Async-Signal-Safety**: Використання системного виклику `write()` замість `printf()` всередині обробника сигналів для запобігання взаємоблокувань (deadlocks).
* **Компіляція**: Makefile з прапорцями `-Wall -Wextra` та обов'язковим лінкуванням бібліотеки потоків `-pthread`.

## Логіка роботи
1. **Ініціалізація**: Налаштування структури sigaction для перехоплення сигналу `SIGSEGV`.
2. **Створення потоків**: Запуск декількох робочих потоків, кожен з яких виконує циклічні обчислення.
3. **Контрольна точка (Checkpoint)**: Використання `sigsetjmp` на початку кожної ітерації для збереження стану стеку та регістрів потоку.
4. **Виникнення збою**: Імітація звернення за нульовою адресою (`NULL pointer dereference`) в одному з потоків.
5. **Обробка та відновлення**:  
   * Ядро надсилає `SIGSEGV`.
   * Обробник (`handler`) інкрементує лічильник ітерацій (щоб пропустити помилковий крок).
   * `siglongjmp` повертає керування в точку збереження, дозволяючи потоку продовжити роботу.

## Технічні характеристики
* **Системні виклики та функції**: `pthread_create()`, `sigaction()`, `sigsetjmp()`, `siglongjmp()`, `write()`.
* **Бібліотеки**: `<pthread.h>`, `<signal.h>`, `<setjmp.h>`, `<unistd.h>`.
* **Механізми**: Багатопоточність (POSIX Threads), обробка сигналів, нелокальні переходи.
* **Середовище**: Linux Mint (на базі Ubuntu 22.04 LTS).

## Очікуваний результат
* Програма запускає декілька потоків, що виводять прогрес ітерацій.
* При досягненні критичної точки (наприклад, ітерація 4 у Потоці 0) стається навмисний збій.
* З'являється повідомлення: `[!] SIGSEGV detected. Skipping failed step and recovering...`.
* Потік успішно відновлюється з ітерації 5 і завершує свою роботу.
* Головний процес виводить: `Main: Finished`

## Висновки
У результаті виконання роботи було опановано методи створення відмовостійких систем.
* **Механізм сигналів**: Вивчено, як ОС сповіщає процеси про критичні помилки доступу до пам'яті.
* **Контекст виконання**: Використання `sigsetjmp/siglongjmp` дозволяє маніпулювати покажчиком команд (IP) та стеком для реалізації логіки відновлення.
* **Безпека в потоках**: Підтверджено важливість розділення контекстів (`__thread`), щоб збій в одному потоці не впливав на стабільність інших та коректно оброблявся індивідуально.
